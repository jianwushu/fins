# Fins 协议技术文档

## 概述

Fins（Factory Interface Network Service）是欧姆龙（Omron）公司开发的一种工业网络通信协议，广泛应用于其 PLC（可编程逻辑控制器）系列产品中。该协议设计用于在自动化控制系统中实现主站与从站之间的高效数据交换，具有良好的实时性和可靠性。Fins 协议支持两种传输层协议：TCP 和 UDP，它们在不同的应用场景下各有优势。TCP 协议提供面向连接的可靠传输，适用于对数据完整性要求较高的场景；UDP 协议则提供无连接的快速传输，适用于对实时性要求较高但对偶尔数据丢失不敏感的场景。

本技术文档旨在为开发 Fins 通信协议库提供全面的技术参考，涵盖协议帧结构、命令集、内存区域寻址、错误处理机制等关键技术细节。文档中的所有内容均基于欧姆龙官方技术规范，并结合实际开发经验进行了详细说明。

## 协议基础

### 协议模型

Fins 协议采用分层架构设计，其底层基于 TCP/IP 或 UDP/IP 协议栈，上层则定义了专门用于工业控制的数据交换格式。从 OSI 模型的角度来看，Fins 协议主要工作在会话层、表示层和应用层，其中会话层负责建立和管理通信连接，表示层负责数据的编码和解码，应用层则定义了具体的命令和响应格式。这种分层设计使得 Fins 协议能够充分利用 TCP/IP 协议栈的成熟特性，同时保持针对工业控制领域的专业性。

在典型的自动化系统架构中，Fins 协议通常用于连接人机界面（HMI）、监控软件、上位机系统与 PLC 控制器。上位机系统作为客户端发起通信请求，PLC 控制器作为服务器端响应请求并执行相应的操作。这种主从式的通信模式简化了系统设计的复杂性，同时也便于实现集中式的监控和管理功能。

### 网络配置

在使用 Fins 协议进行通信之前，需要正确配置网络参数。PLC 控制器通常具有一个唯一的 IP 地址，用于在网络中标识该设备。Fins 协议引入了节点地址的概念，节点地址是一个 8 位的标识符，用于在同一网络中区分不同的 PLC 设备。通信端口方面，Fins TCP 和 Fins UDP 均使用端口 9600 进行通信。

对于 UDP 模式下的广播通信，发送方可以使用特定的广播地址将消息发送到网络中的所有设备。这种方式在设备发现和自动配置场景中非常有用。广播地址的设置需要根据实际网络环境进行配置，常见的广播地址格式为「IP地址.255」或「255.255.255.255」。

## Fins TCP 协议

### 协议特点

Fins TCP 是基于 TCP 传输层协议的 Fins 通信实现，它继承了 TCP 协议的所有优点，包括面向连接、可靠传输、流量控制和拥塞控制等特性。在 Fins TCP 模式下，通信双方需要首先建立 TCP 连接，然后才能进行数据交换。连接建立过程遵循标准的 TCP 三次握手流程，一旦连接建立成功，双方就可以通过该连接进行双向通信。

Fins TCP 的主要优势在于其可靠性和稳定性。由于 TCP 协议保证数据包的顺序传输和完整性校验，因此开发者无需在应用层实现额外的可靠性机制。此外，TCP 协议的流量控制和拥塞控制机制能够有效地避免网络拥塞，确保通信的平稳进行。这些特性使得 Fins TCP 特别适用于对数据完整性要求较高的场景，如关键设备的状态监控、重要参数的配置下发等。

### TCP 帧结构

Fins TCP 帧使用专用的帧头格式，包含以下字段：

| 字段名称 | 长度（字节） | 说明 |
|---------|-------------|------|
| 魔数 | 4 | 固定值 0x46494E53（ASCII "FINS"） |
| 长度 | 4 | 整个帧的长度（网络字节序），包括头部 |
| 错误码 | 4 | 响应帧中使用，表示执行结果 |
| 客户端节点号 | 4 | 发送方的节点标识 |
| 服务器节点号 | 4 | 接收方的节点标识 |

Fins TCP 帧的魔数字段是一个固定的 4 字节值「FINS」，用于在 TCP 流中识别 Fins 帧的起始位置。长度字段表示整个帧的字节数，包括魔数、长度字段本身以及后续的所有数据。错误码字段在请求帧中通常为 0，在响应帧中用于表示命令执行的状态。客户端节点号和服务器节点号用于在多设备环境中标识通信双方。

在 TCP 模式下，由于 TCP 是流协议而非消息协议，接收方可能会将发送方的多个 Fins 帧合并为一个 TCP 数据包，或者将单个 Fins 帧拆分为多个 TCP 数据包。因此，在接收数据时需要进行帧边界识别。识别方法是：首先读取 4 字节魔数，确认是 Fins 帧后，再读取 4 字节长度字段，根据长度读取完整的帧数据。

### 连接管理

Fins TCP 连接的管理涉及连接建立、数据传输和连接关闭三个阶段。在连接建立阶段，客户端需要知道 PLC 设备的 IP 地址和监听端口（默认为 9600），然后使用标准的 socket API 发起连接请求。连接建立成功后，双方进入数据传输阶段，可以开始交换 Fins 协议数据帧。

在数据传输阶段，需要注意 TCP 协议的特性对 Fins 通信的影响。常用的帧边界识别方法是使用「读取魔数 + 读取长度字段 + 读取指定长度数据」的三步读取方法。

连接关闭阶段遵循 TCP 的四次挥手流程。在关闭连接之前，确保所有待处理的 Fins 帧都已经得到响应，避免数据丢失。如果检测到连接异常断开，应用层应该实现重连机制，以恢复通信能力。

### 数据传输流程

Fins TCP 的典型数据传输流程包括请求发送、响应接收和结果处理三个步骤。在请求发送阶段，客户端根据要执行的操作构建相应的 Fins 请求帧，并通过已建立的 TCP 连接发送出去。请求帧的构建需要遵循 Fins 协议的数据格式规范，包括正确设置命令码、地址信息和参数数据等。

发送请求后，客户端进入等待响应状态。由于 TCP 是可靠传输协议，发送的数据包最终会被对方接收，但响应的返回时间取决于 PLC 的处理速度和当前网络状况。因此，在实现时需要设置合理的超时机制，避免因 PLC 无响应导致的应用挂起。超时时间的设置应该考虑到实际应用场景的需求，既不能太短导致误判，也不能太长影响用户体验。

收到响应后，客户端需要解析响应帧，提取状态码和返回数据。状态码为 0 表示命令执行成功，否则表示发生了错误，需要根据错误码进行相应的错误处理。

## Fins UDP 协议

### 协议特点

Fins UDP 是基于 UDP 传输层协议的 Fins 通信实现，它采用无连接的通信模式，不需要事先建立连接即可发送数据。这种设计使得 Fins UDP 具有较低的通信延迟，适用于对实时性要求较高的场景。由于 UDP 协议没有连接建立和维护的开销，因此在需要频繁进行小数据量通信的应用中，Fins UDP 的效率可能更高。

然而，Fins UDP 的无连接特性也带来了一些局限性。首先，UDP 协议不保证数据包的到达和顺序，可能出现丢包、乱序或重复等问题，因此需要在应用层实现相应的可靠性机制。其次，UDP 协议没有流量控制和拥塞控制，在高负载情况下可能出现数据丢失。这些局限性使得 Fins UDP 更适用于对实时性要求高但对偶尔数据丢失不敏感的场景，如实时状态监控、周期性的数据采集等。

### UDP 帧头详解

Fins UDP 帧使用经典的 Fins 帧头格式，包含以下字段：

| 字段名称 | 长度（字节） | 说明 |
|---------|-------------|------|
| ICF | 1 | 信息控制字段 |
| RSV | 1 | 保留字段，固定为 0x00 |
| GCT | 1 | 网关计数 |
| DNA | 1 | 目标网络地址 |
| DA1 | 1 | 目标节点地址 |
| DA2 | 1 | 目标单元地址 |
| SNA | 1 | 源网络地址 |
| SA1 | 1 | 源节点地址 |
| SA2 | 1 | 源单元地址 |
| SID | 1 | 服务 ID |
| 命令码 | 2 | 通信命令类型 |
| 参数数据 | 变长 | 命令参数或响应数据 |

### 地址参数说明

地址参数（DNA、DA1、DA2、SNA、SA1、SA2）是 Fins UDP 帧中用于路由和寻址的关键字段，这些参数需要根据通信双方的配置动态赋值。

**目标地址参数（DNA、DA1、DA2）**：

| 字段 | 有效值 | 说明 |
|-----|-------|------|
| DNA（目标网络地址） | 0x00-0xFF | 0x00 表示本地网络，其他值用于跨网络通信 |
| DA1（目标节点地址） | 0x00-0xFF | 0x00 表示广播地址（发送给所有节点），0x01-0xFE 表示特定节点地址 |
| DA2（目标单元地址） | 0x00-0xFF | 0x00 表示 CPU 单元，0xFE 表示内置 Ethernet 端口，0x01-0x0F 表示扩展单元 |

**源地址参数（SNA、SA1、SA2）**：

| 字段 | 有效值 | 说明 |
|-----|-------|------|
| SNA（源网络地址） | 0x00-0xFF | 通常设置为 0x00（本地网络） |
| SA1（源节点地址） | 0x00-0xFF | 发送方的节点地址，在 UDP 模式下可自由设置 |
| SA2（源单元地址） | 0x00-0xFF | 发送方的单元地址，通常设置为 0x00 |

**地址参数赋值规则**：

```go
// 本地通信示例
frame.DNA = 0x00    // 本地网络
frame.DA1 = 0x64    // 目标节点地址为 100（PLC 的 Fins 节点号）
frame.DA2 = 0x00    // CPU 单元

frame.SNA = 0x00    // 本地网络
frame.SA1 = 0x01    // 客户端节点地址（可自定义）
frame.SA2 = 0x00    // 客户端单元

// 广播通信示例
frame.DNA = 0x00    // 本地网络
frame.DA1 = 0x00    // 广播地址
frame.DA2 = 0x00    // CPU 单元
```

**获取 PLC 节点地址的方法**：

在 Fins UDP 模式下，通常需要先发送一个「节点地址请求」帧来获取 PLC 的 Fins 节点地址。请求帧使用命令码 0x0501（控制器读取），配合特定的参数格式可获取 PLC 的网络配置信息，包括其 Fins 节点地址。

### SID（Service ID）详解

SID（Service ID，服务 ID）是 Fins UDP 协议中用于匹配请求和响应的关键字段，它在应用层实现类似 TCP 连接会话的功能。

**SID 的作用**：

1. **请求-响应匹配**：发送方在请求帧中设置 SID，响应帧中会返回相同的 SID 值，用于确认响应与请求的对应关系。

2. **会话隔离**：在多客户端或多请求场景下，SID 用于区分不同的会话，避免响应混淆。

3. **重传识别**：当请求超时时，发送方重新发送请求，此时可以保持相同的 SID 或使用新的 SID。

**SID 取值范围**：0x00-0xFF（8 位无符号整数）

### SID 模式配置

协议库支持两种 SID 模式，可通过配置开关进行选择：

| 模式 | 配置值 | 说明 | 适用场景 |
|-----|-------|------|---------|
| 固定模式 | SIDFixed | 所有请求使用固定的 SID 值（默认 0x00） | 单客户端简单场景 |
| 递增模式 | SIDIncrement | 每次请求 SID 递增，实现伪序列号 | 多请求并发场景 |

**SID 模式配置示例**：

```go
// 配置 SID 模式
type SIDMode int

const (
    SIDFixed    SIDMode = 0  // 固定模式
    SIDIncrement SIDMode = 1 // 递增模式
)

type FinsClientConfig struct {
    SIDMode        SIDMode  // SID 模式选择
    FixedSID       byte     // 固定模式下的 SID 值
    StartSID       byte     // 递增模式的起始值
    MaxSID         byte     // SID 循环最大值
}

type FinsClient struct {
    config        *FinsClientConfig
    sequenceNo    uint16    // 递增序列号（用于 SID 溢出处理）
    currentSID    byte      // 当前使用的 SID
}

// 初始化客户端时配置 SID 模式
func NewFinsClient(ip string, port int, config *FinsClientConfig) *FinsClient {
    if config == nil {
        config = &FinsClientConfig{
            SIDMode:    SIDFixed,   // 默认使用固定模式
            FixedSID:   0x00,       // 默认 SID 为 0
            StartSID:   0x00,
            MaxSID:     0xFF,
        }
    }
    return &FinsClient{
        config: config,
        sequenceNo: uint16(config.StartSID),
        currentSID: config.FixedSID,
    }
}

// 获取下一个 SID 值
func (c *FinsClient) getNextSID() byte {
    switch c.config.SIDMode {
    case SIDFixed:
        // 固定模式：始终返回配置的固定 SID
        return c.config.FixedSID
    case SIDIncrement:
        // 递增模式：每次调用后递增
        c.sequenceNo++
        if uint16(c.config.MaxSID) == 0xFF && c.sequenceNo%256 == 0 {
            c.sequenceNo = uint16(c.config.StartSID)
        }
        return byte(c.sequenceNo)
    default:
        return c.config.FixedSID
    }
}

// 发送请求时设置 SID
func (c *FinsClient) SendRequest(cmd uint16, params []byte) (*FinsResponse, error) {
    sid := c.getNextSID()
    
    frame := &FinsUDPFrame{
        ICF: 0x02,        // 请求响应模式
        RSV: 0x00,
        GCT: 0x02,
        DNA: 0x00,        // 本地网络
        DA1: c.serverNode, // 目标节点（动态获取）
        DA2: 0x00,        // CPU 单元
        SNA: 0x00,        // 本地网络
        SA1: c.localNode,  // 本地节点
        SA2: 0x00,        // 客户端单元
        SID: sid,         // 根据模式获取 SID
        Command: cmd,
        Data: params,
    }
    
    response, err := c.sendAndWait(frame, sid)
    return response, err
}
```

**使用场景建议**：

| 场景 | 推荐模式 | 原因 |
|-----|---------|------|
| 单客户端、单请求顺序执行 | 固定模式 | 简单可靠，无需维护状态 |
| 单客户端、并发请求 | 递增模式 | 避免响应混淆，便于匹配 |
| 多客户端共享连接池 | 递增模式（每客户端独立 SID 范围） | 隔离不同客户端的会话 |
| PLC 只接受特定 SID | 固定模式 | 按 PLC 要求设置固定值 |

**SID 与 TCP 连接的关系**：

在 Fins TCP 模式下，由于 TCP 本身提供了连接级别的会话管理，SID 通常固定为 0x00。但在某些实现中，SID 仍可用于在同一 TCP 连接上区分不同的请求流。

### 广播通信

Fins UDP 支持广播通信模式，允许发送方将消息发送到网络中的所有设备。在广播模式下，发送方使用特定的广播地址作为目标地址，UDP 数据包会被网络中的所有设备接收。这种通信模式在设备发现、自动配置和批量控制等场景中非常有用。

使用广播通信时，需要注意以下几点：首先，广播通信会产生网络流量，影响网络性能，因此应该尽量减少广播消息的频率和数据量；其次，广播消息可能会被防火墙或路由器过滤，需要确保网络配置允许广播流量通过；最后，接收方需要正确处理广播消息，避免响应不必要或恶意的广播请求。

### 可靠性机制

由于 UDP 协议本身不提供可靠性保证，在使用 Fins UDP 时需要在应用层实现可靠性机制。常用的可靠性机制包括：超时重传、序列号确认和心跳检测。

超时重传机制要求发送方在发送请求后启动一个定时器，如果在规定时间内没有收到响应，则重新发送请求。定时器的超时时间应该根据网络状况动态调整，可以使用指数退避算法避免网络拥塞。序列号确认机制要求每个请求都有一个唯一的序列号，接收方在响应中携带相同的序列号，以便发送方确认响应是否对应自己的请求。心跳检测机制定期发送心跳包，用于检测连接状态和保持会话活跃。

## TCP 与 UDP 帧结构对比

| 特性 | Fins TCP | Fins UDP |
|-----|---------|----------|
| 帧起始标识 | 4 字节魔数 "FINS" | 1 字节 ICF 字段 |
| 帧头长度 | 20 字节 | 10 字节 |
| 连接方式 | 面向连接 | 无连接 |
| 可靠性 | TCP 保证 | 需要应用层实现 |
| 地址参数 | 节点号（4 字节） | DNA、DA1、DA2、SNA、SA1、SA2（6 字节） |
| 会话管理 | TCP 连接 | SID（1 字节），支持固定/递增模式 |
| 数据边界 | 通过长度字段识别 | UDP 本身是消息协议 |
| 适用场景 | 重要数据传输 | 实时性要求高的场景 |

## 命令详解

### 内存读取命令（0x0101）

内存读取命令用于从 PLC 的内存区域中读取数据。该命令是 Fins 协议中最常用的命令之一，广泛应用于状态监控和数据采集等场景。

**请求帧参数格式**：

| 字段 | 长度 | 说明 |
|-----|------|------|
| 内存区域代码 | 1 | 指定要读取的内存区域 |
| 起始地址 | 3 | 目标起始地址（2 字节地址 + 1 字节位号，位操作时使用） |
| 数据类型 | 1 | 指定读取数据的类型 |
| 数据长度 | 2 | 要读取的数据项数 |

**内存区域代码**：

| 代码 | 内存区域 | 说明 |
|-----|---------|------|
| 0x30 | CIO 区 | 输入输出继电器 |
| 0x31 | WR 区 | 工作继电器 |
| 0x32 | HR 区 | 保持继电器 |
| 0x33 | TC 区 | 定时器/计数器完成标志 |
| 0x34 | A 区 | 辅助继电器 |
| 0x82 | D 区 | 数据寄存器 |
| 0x89 | T 区 | 定时器当前值 |
| 0x8C | C 区 | 计数器当前值 |

**数据类型**：

| 代码 | 数据类型 | 说明 |
|-----|---------|------|
| 0x00 | 位（Bit） | 读取单个位 |
| 0x01 | 字（Word） | 读取 16 位数据 |

**响应帧格式**：

响应帧包含状态码和读取到的数据。如果状态码为 0x0000，表示命令执行成功，数据字段中包含读取到的实际数据；否则，数据字段中包含错误码。

### 内存写入命令（0x0102）

内存写入命令用于向 PLC 的内存区域中写入数据。该命令适用于参数设置、控制指令下发等场景。

**请求帧参数格式**：

| 字段 | 长度 | 说明 |
|-----|------|------|
| 内存区域代码 | 1 | 指定要写入的内存区域 |
| 起始地址 | 3 | 目标起始地址 |
| 数据类型 | 1 | 指定写入数据的类型 |
| 数据长度 | 2 | 要写入的数据项数 |
| 写入数据 | 变长 | 要写入的具体数据 |

在执行写入操作时需要特别注意安全性。由于写入操作会直接修改 PLC 的内存内容，错误的写入可能导致设备故障或生产事故。因此，在实际应用中应该实现权限控制机制，只允许授权的用户或系统执行写入操作。此外，建议在写入前先进行读取验证，确保写入的数据符合预期。

写入命令支持批量写入功能，可以在一次命令中写入多个连续的字。批量写入可以减少通信次数，提高写入效率。但需要注意目标设备的限制，不同型号的 PLC 可能对单次写入的数据量有不同的限制。

## 内存区域寻址

### 地址格式

内存地址由内存区域代码和偏移地址两部分组成。偏移地址是一个 16 位或 32 位的无符号整数，表示相对于内存区域起始地址的字节偏移量。对于位操作，还需要指定位号（0-15）。

例如：
- 地址「D100」表示 D 区域中第 100 号寄存器（字地址）
- 地址「W0.05」表示 W 区域第 0 号寄存器的第 5 位

在构建 Fins 帧时，需要将这种易读的地址格式转换为协议规定的二进制格式。地址的表示方式因 PLC 型号不同而有所差异，需要参考相应设备的编程手册。

### 连续地址范围

对于连续的内存区域，可以使用范围地址表示多个连续的内存单元。范围地址在批量读写操作中非常有用，可以显著减少通信次数。

例如：
- 「D100-D199」表示从 D100 到 D199 的 100 个寄存器
- 「CIO0-CIO99」表示从 CIO0 到 CIO99 的 100 个位

### 异常处理策略

在协议库的实现中，应该建立完善的异常处理机制，确保在各种异常情况下能够正确响应并恢复。常见的异常处理策略包括：

1. **连接异常处理**：当检测到连接断开时，尝试自动重连，并通知上层应用连接状态变化。

2. **响应超时处理**：当请求超时时，根据业务需求决定是否重试以及重试次数。对于关键操作，建议设置最大重试次数限制，避免无限重试导致的问题。

3. **数据校验失败处理**：当检测到数据校验错误时，丢弃当前数据帧，请求重发或上报错误。

4. **资源释放处理**：在任何异常情况下，确保已分配的资源（如内存、文件句柄等）能够正确释放，避免资源泄漏。

## 实现要点

### Socket 编程注意事项

在实现 Fins 协议库时，底层通信通常使用 socket 编程接口。对于 Fins TCP，需要注意以下几点：

1. **连接管理**：建立连接时处理各种可能的错误情况，如连接超时、连接被拒绝等。实现连接池机制，避免频繁创建和销毁连接带来的开销。

2. **数据接收**：由于 TCP 是流协议，需要实现帧边界识别逻辑。推荐使用「读取魔数 + 读取长度字段 + 读取指定长度数据」的三步读取方法。

3. **并发处理**：如果需要支持多客户端并发访问，应该使用多线程或多路复用技术，如 select、poll 或 epoll。

对于 Fins UDP，需要注意以下几点：

1. **绑定地址**：服务器端需要绑定到特定的地址和端口，以便接收客户端请求。客户端则可以绑定到任意可用端口。

2. **缓冲区管理**：UDP 数据包有大小限制（通常为 1472 字节），超出限制的数据包可能被丢弃或分片。建议在应用层实现数据分片和重组机制。

3. **广播设置**：使用广播功能时，需要显式设置 socket 选项 SO_BROADCAST。

4. **动态地址管理**：实现地址参数动态赋值机制，在通信过程中根据需要更新 DNA、DA1、DA2、SNA、SA1、SA2 的值。

5. **SID 模式支持**：根据配置选择固定模式或递增模式生成 SID 值。

### 帧解析与构建

帧解析是协议库的核心功能之一，需要高效且正确地处理各种输入数据。

**Fins TCP 帧解析步骤**：

1. 读取 4 字节魔数，确认为 "FINS"
2. 读取 4 字节长度字段，获取帧总长度
3. 根据长度读取剩余数据
4. 解析错误码、节点号和应用层数据

**Fins UDP 帧解析步骤**：

1. 解析 10 字节帧头（ICF、RSV、GCT、DNA、DA1、DA2、SNA、SA1、SA2、SID）
2. 读取 2 字节命令码
3. 验证 SID 是否匹配当前会话（根据 SID 模式）
4. 解析应用层参数或响应数据

帧构建过程是解析的逆过程，需要按照协议规定的格式依次填充各个字段。

### 线程安全设计

如果协议库需要支持多线程环境，必须确保所有共享数据的访问都是线程安全的。常用的线程安全设计方法包括：

1. **互斥锁**：对共享资源的访问使用互斥锁保护，确保同一时间只有一个线程能够访问。

2. **无状态设计**：尽量设计无状态的接口，避免在库内部维护共享状态。

3. **连接池管理**：如果维护连接池，需要使用线程安全的数据结构管理连接。

4. **SID 生成**：在多线程环境下，SID 的生成需要加锁保护，避免竞争条件。

## 应用示例

### 读取单个寄存器

以下示例展示了如何使用 Fins UDP 协议读取 D100 寄存器的值：

```go
// 配置客户端（使用固定 SID 模式）
config := &FinsClientConfig{
    SIDMode:    SIDFixed,    // 固定模式
    FixedSID:   0x00,        // SID=0
}

client := NewFinsClient("192.168.1.10", 9600, config)
client.SetLocalNode(0x01)      // 设置本地节点地址
client.SetServerNode(0x64)     // 设置服务器节点地址（需从 PLC 获取）

// 构建读取请求
params := []byte{
    0x82,             // D区
    0x00, 0x64,       // 地址 100（低字节在前）
    0x00,             // 字类型
    0x00, 0x01,       // 1个字
}

// 发送请求并解析响应
response, err := client.ReadMemory(0x0101, params)
if err != nil {
    log.Fatalf("读取失败: %v", err)
}

// 解析响应数据
if response.Status == 0 {
    data := response.Data[2:] // 跳过 2 字节状态码
    value := uint16(data[0])<<8 | uint16(data[1])
    fmt.Printf("D100 = %d\n", value)
}
```

### 写入多个寄存器

以下示例展示了如何使用 Fins UDP 协议向 D200-D204 写入 5 个字的数据：

```go
// 配置客户端（使用递增 SID 模式）
config := &FinsClientConfig{
    SIDMode:    SIDIncrement, // 递增模式
    StartSID:   0x01,         // 起始 SID=1
    MaxSID:     0xFF,
}

client := NewFinsClient("192.168.1.10", 9600, config)
client.SetLocalNode(0x01)
client.SetServerNode(0x64)

// 构建写入数据
writeData := []byte{
    0x82,             // D区
    0x00, 0xC8,       // 地址 200
    0x00,             // 字类型
    0x00, 0x05,       // 5个字
    0x00, 0x01,       // 数据 1
    0x00, 0x02,       // 数据 2
    0x00, 0x03,       // 数据 3
    0x00, 0x04,       // 数据 4
    0x00, 0x05,       // 数据 5
}

// 发送写入请求（每次调用 SID 会自动递增）
response, err := client.WriteMemory(0x0102, writeData)
if err != nil {
    log.Fatalf("写入失败: %v", err)
}

if response.Status == 0 {
    fmt.Println("写入成功")
}
```

### SID 会话管理示例

```go
type FinsSession struct {
    client      *FinsClient
    pendingReq  map[byte]*PendingRequest
    mutex       sync.Mutex
    timeout     time.Duration
}

type PendingRequest struct {
    SID       byte
    Request   []byte
    CreatedAt time.Time
    Response  chan *FinsResponse
}

func (s *FinsSession) SendWithSID(params []byte) (*FinsResponse, error) {
    // 根据配置的 SID 模式获取 SID
    sid := s.client.getNextSID()
    
    req := &PendingRequest{
        SID:       sid,
        Request:   params,
        CreatedAt: time.Now(),
        Response:  make(chan *FinsResponse, 1),
    }
    
    s.mutex.Lock()
    s.pendingReq[sid] = req
    s.mutex.Unlock()
    
    // 发送请求
    err := s.client.SendFrame(sid, params)
    if err != nil {
        return nil, err
    }
    
    // 等待响应或超时
    select {
    case resp := <-req.Response:
        return resp, nil
    case <-time.After(s.timeout):
        s.mutex.Lock()
        delete(s.pendingReq, sid)
        s.mutex.Unlock()
        return nil, ErrTimeout
    }
}

// 处理接收到的响应
func (s *FinsSession) HandleResponse(sid byte, data []byte) {
    s.mutex.Lock()
    req, exists := s.pendingReq[sid]
    s.mutex.Unlock()
    
    if exists {
        resp := &FinsResponse{
            SID:    sid,
            Status: uint16(data[0])<<8 | uint16(data[1]),
            Data:   data[2:],
        }
        req.Response <- resp
    }
}
```
